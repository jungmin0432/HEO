# 검증 설계 (Validation Design)

작성 시점: 이 문서는 held-out 검증 스크립트(`validate_holdout.py`)를 실행하기 **이전에** 커밋한다.
목적: 결과를 본 뒤 유리한 지표를 고르는 것을 막기 위해, 성공/실패 판정 기준을 실행 전에 고정한다.

## 왜 기존 벤치마크(`benchmark_results.md`)를 그대로 쓰지 않는가

`benchmark_baseline.py`의 120+18개 시나리오는 이미 결과를 확인한 뒤 이 문서를 쓰고 있으므로,
그 숫자를 "검증"이라고 부르면 사후 선택(post-hoc cherry-picking) 문제를 피할 수 없다.
따라서 아래 held-out 시나리오는 그 벤치마크가 다루지 않은 축(영업시간이 달라지는 시간대,
무장애 경로 조건)만 새로 구성해서, 아직 결과를 보지 않은 상태로 기준부터 정한다.

## Held-out 시나리오 구성 (아직 실행 전)

1. **시간대 안정성 세트**: 평일 오전(10:30) · 평일 저녁(21:00) · 토요일 오후 · 일요일 오후 4개 시각 x
   전체 6개 게이트 x 목적 3종(look, make, rest) x 30분 고정 = 72건. 요일별로 영업 중인 점포 구성이
   달라지므로(WEEKDAYS만 여는 점포는 일요일에 빠짐) 이전 벤치마크(평일 15:00 고정)와 겹치지 않는다.
2. **접근성 세트**: 전체 6개 게이트 x 시간 4종(10/30/60/90) x `accessible=True` 고정, 목적 look = 24건.
   이전 벤치마크는 전부 `accessible=False`였으므로 완전히 새로운 조건이다.

총 96건, 이전 138건(120+18)과 시나리오가 겹치지 않는다.

## 사전 등록 성공 기준 (Pass/Fail)

| # | 기준 | 판정 방법 | 근거 |
|---|---|---|---|
| A | 시간 활용도 우위 | held-out 전체에서 현재 엔진의 평균 시간활용률이 그리디 baseline보다 **+10%p 이상** 높아야 통과 | 방문자의 한정된 시간을 그리디보다 의미 있게 더 채워야 알고리즘을 쓰는 이유가 성립 |
| B | 거래 연결 우위 | 시간대 세트(목적지 미지정)에서 혼합 달성률(신규공간+기존점포 동시 포함)이 baseline보다 **+5%p 이상** 높아야 통과 | 사업모델의 핵심(체험→기존 점포 거래 연결)이 baseline보다 확실히 나아야 함 |
| C | 보행 부담 가드레일 | 평균 정류장당 도보거리 증가폭이 baseline 대비 **+100%(2배) 이내**여야 통과. 초과하면 실패로 기록하고 한계로 명시 | 시간·거래를 최적화해도 과도하게 걷게 만들면 실사용에서 이탈 위험 |
| D | 시간대 안정성 | 4개 시간대 각각에서 현재 엔진의 코스 산출 **성공률 90% 이상** | 영업시간이 바뀌는 상황에서도 안정적으로 코스를 내야 실사용 가능 |
| E | 접근성 최소 기준 | `accessible=True` 조건에서 현재 엔진 성공률 **80% 이상** | 우회 경로 때문에 다소 낮아도 되지만, 무장애 이용자를 사실상 배제해선 안 됨 |

**종합 판정 규칙**: A·B·D·E는 전부 통과해야 "검증 통과"로 표시한다. C는 가드레일이므로 실패해도
전체 결과를 무효화하지 않되, 반드시 한계(6번 기준)에 그대로 기록한다.
