# AI 시간제작소 2차 서비스 구상 보고서

## 1. 서비스 정의

`현재 사진과 과거 기록을 연결하기`는 사용자가 을지로 지하상가에서 촬영한 현재 사진을 올리면, 공개 기록 사진 후보를 순위화하고 각 후보가 왜 제시되었는지 설명하는 기능이다.

이 기능은 동일 장소를 자동 확정하거나 과거 사진의 사실성을 보증하지 않는다. 화면에는 `후보 검색 점수`와 근거를 표시하며, 근거가 부족하거나 충돌하면 `판정 보류`로 반환한다.

## 1.1 문제 정의와 현재 기준선

### 누구의 문제인가

- 을지로 지하상가를 지나며 현재 장면을 기록하고, 이 장소의 과거 자료를 연결해 보고 싶은 방문자
- 공개 기록 사진을 보유하지만, 사진의 정확한 현재 위치를 일일이 수작업으로 확인해야 하는 서비스 운영자

### 무엇이 얼마나 어려운가

현재 프로젝트에 보관된 공개 기록 사진은 **56장, 3개 기록 묶음**이다. 사진별 공식 출처는 일부 확보되어 있으나, 개별 사진의 검증된 좌표·적용 반경·랜드마크 태그는 아직 구조화되어 있지 않다. 따라서 현 상태에서는 한 장의 현재 사진에 대해 최대 56장을 육안으로 비교해야 하며, 위치 근거를 이용해 후보군을 줄일 수 있는 비율은 **0/56장**이다.

이 수치는 기능 도입 전 기준선이다. 이후 메타데이터를 채우면 `좌표 또는 랜드마크 근거를 가진 기록 비율`, `검색 전 후보 수`, `상위 5개 후보에 검증 관계가 포함되는 비율`, `수작업 후보 확인 시간`을 같은 방식으로 다시 측정한다.

### 기존 방식이 깨지는 지점

1. 키워드 검색은 기록 제목에 없는 통로, 기둥, 천장, 출입구 정보를 찾기 어렵다.
2. 이미지 유사도만 쓰면 색, 조명, 사람, 상점 변화에 의해 다른 공간이 상위에 오를 수 있다.
3. GPS만 쓰면 지하 환경의 위치 오차와 사진의 부정확한 과거 촬영 지점 때문에 잘못된 제외가 생긴다.

따라서 이 서비스는 어느 하나의 신호로 정답을 단정하지 않고, 서로 다른 근거를 결합해 `후보 탐색`을 지원한다.

## 2. 사용자 흐름

1. 사용자가 현재 사진을 업로드한다.
2. 위도, 경도, GPS 정확도(m)를 직접 입력한다. 지하 GPS의 오차를 고려해 선택 입력으로 둔다.
3. 주변 건물, 역 출입구, 통로명, 상가 구역을 자유 텍스트로 입력한다.
4. 시스템이 위치와 태그로 공개 기록 사진 후보군을 먼저 좁힌다.
5. DINOv2와 CLIP의 이미지 특징으로 후보를 재정렬한다.
6. 상위 후보에 한해 국소 특징 매칭을 수행하고, 후보별 `위치 근접`, `장소 단서`, `시각적 구조`, `판정 한계`를 제시한다.
7. 사용자가 후보를 선택하면 해당 기록을 기준으로 사진 복원 또는 비교 화면으로 넘어간다.

## 3. 모델과 점수 결합

### 3.1 후보 생성

- GPS: 기록 사진별 중심 좌표와 적용 반경을 활용한 거리 감쇠 점수. GPS 정확도가 낮거나 누락되면 가중치를 낮춘다.
- 장소 단서: 입력 문장과 기록의 역, 출입구, 주변 건물, 구역 태그를 비교한다. 후보의 공식 메타데이터에 없는 장소 단서는 근거로 표시하지 않는다.
- 메타데이터가 부족한 기록은 후보에서 배제하지 않지만, 위치 근거 점수는 부여하지 않는다.

### 3.2 이미지 검색 앙상블

| 구성 | 역할 | 저장 산출물 |
| --- | --- | --- |
| DINOv2 ViT-S/14 | 기둥, 천장선, 통로 폭 등 이미지 구조 중심의 후보 검색 | L2 정규화 임베딩, 마지막 블록 CLS-to-patch attention 요약 |
| CLIP ViT-B/32 | 전반적 장면 유사도와 사진-텍스트 의미 연결 보조 | L2 정규화 이미지 임베딩, 기록 사진 간 cosine 행렬 |
| DISK + LightGlue | 상위 3~5개 후보의 대응점 검증 | 대응점 수, RANSAC inlier 비율, 검증 실패 사유 |

CLIP은 이미지와 텍스트 특징을 같은 잠재공간에 투영하고 내적으로 유사도를 계산한다. DINOv2는 이미지별 pooling 출력을 제공하므로 이미지 검색용 특징 벡터로 사용한다. LightGlue는 국소 특징의 대응점을 찾는 사전학습 매처다.

출처: [Hugging Face CLIP 문서](https://huggingface.co/docs/transformers/model_doc/clip), [Hugging Face DINOv2 문서](https://huggingface.co/docs/transformers/model_doc/dinov2), [LightGlue 공식 저장소](https://github.com/cvg/LightGlue).

### 3.3 점수 정규화와 결합

모델의 원시 cosine 점수는 모델마다 분포가 달라 직접 평균하지 않는다. 각 질의에서 후보군 내 백분위 또는 검증 세트 기반 z-score로 정규화한 뒤 결합한다.

초기 가설은 다음과 같으며, 구현 전 실험으로 확정한다.

```text
GPS 신뢰 가능: 0.35 geo + 0.25 dino + 0.15 clip + 0.15 landmark + 0.10 local-feature
GPS 누락/저신뢰: 0.40 dino + 0.25 clip + 0.20 landmark + 0.15 local-feature
```

화면에는 `후보 검색 점수 0~100`과 근거 항목을 표시한다. 이 수치는 동일 지점일 확률이나 모델 정확도가 아니다.

## 4. 데이터 인덱스

각 기록 사진은 다음 정보를 갖는다.

```json
{
  "asset_id": "archive-image-id",
  "asset_path": "history_photo/.../image.jpg",
  "archive_source_url": "https://archives.seoul.go.kr/...",
  "year": 1983,
  "zone": "euljiro-3ga",
  "location": {"latitude": null, "longitude": null, "radius_m": null, "source_url": null},
  "landmark_tags": ["을지로3가역", "지하연결통로"],
  "verification_level": "REFERENCE"
}
```

현재 보유 사진은 공개 기록 사진 56장이다. 좌표, 반경, 랜드마크 태그는 원문 출처 또는 공공 지도 자료가 확인된 뒤에만 채운다. 추정 좌표를 사실값으로 저장하지 않는다.

## 5. 사전 계산 캐시

`scripts/build_archive_feature_cache.py`는 공개 기록 사진을 읽어 다음 파일을 생성한다.

- `cache/location_matching/manifest.json`: 파일 해시, 경로, 모델 버전, 이미지 순서
- `clip_embeddings.npy`, `dino_embeddings.npy`: 재사용 가능한 이미지 임베딩
- `clip_similarity.npy`, `dino_similarity.npy`: 기록 사진 간 cosine 유사도 행렬
- `dino_attention_maps.npy`: 마지막 블록 CLS-to-patch attention 맵
- `attention_summary.json`: 최대 attention, 엔트로피, 상위 patch 집중도

사진이 추가되거나 모델 버전이 바뀐 경우에만 캐시를 다시 생성한다. API와 UI는 이후 단계에서 이 파일을 읽고, 매 요청마다 모든 기록 사진의 특징을 다시 계산하지 않는다.

## 6. 구현 전 실험 설계

### 6.0 평가 기준과 사전 성공 조건

부트캠프 평가 기준에 따라 모델을 실행하기 전에 비교군과 성공·실패 기준을 먼저 고정한다. 아래 목표는 결과를 본 뒤 변경하지 않으며, 데이터가 늘어나면 새 버전의 검증 세트를 별도로 만든다.

| 검증 질문 | 성공 기준 | 실패 또는 보류 기준 |
| --- | --- | --- |
| 이미지 앙상블이 단일 모델보다 나은가 | 홀드아웃에서 Recall@3가 최선의 단일 모델보다 10%p 이상 높음 | 개선폭 10%p 미만이면 단일 모델 또는 가중치 재설계 |
| 위치·장소 단서가 실제로 도움이 되는가 | 이미지 앙상블 대비 nDCG@5가 0.05 이상 상승하고 무관 후보 Top-1 비율이 증가하지 않음 | 순위 향상 없이 오추천이 늘면 해당 단서 가중치 축소 |
| 강한 추천이 믿을 만한가 | 사전 정의한 상위 점수 구간의 동일 지점/동일 구역 적중률 80% 이상 | 80% 미만이면 해당 구간을 `판정 보류`로 변경 |
| 시현 환경에서 쓸 수 있는가 | 캐시가 있는 상태의 p95 응답시간 8초 이하, API 오류율 1% 이하 | 기준 미달이면 작은 모델 또는 상위 후보만 국소 검증 |
| 근거 문장이 출처를 갖는가 | 화면에 제시되는 위치·연도·장소 근거의 출처 연결률 100% | 출처 없는 근거는 화면에서 제거 |

정답 라벨이 아직 충분하지 않은 현 단계에서는 이 기준을 `실험 설계`로 보관한다. 20장 이상의 현재 사진 질의와 독립 검토 라벨이 확보되기 전에는 모델 우위나 정확도를 주장하지 않는다.

### 6.1 정답 라벨

현재 사진과 기록 사진의 관계를 `동일 지점`, `동일 구역`, `주변 참고`, `무관` 네 단계로 라벨링한다. 동일 촬영 지점이 공식적으로 확인되지 않은 자료에는 `동일 지점` 라벨을 부여하지 않는다.

### 6.2 비교군

1. DINOv2만 사용
2. CLIP만 사용
3. DINOv2 + CLIP 정규화 앙상블
4. 이미지 앙상블 + 장소 단서
5. 이미지 앙상블 + 장소 단서 + GPS
6. 5번의 상위 후보에 LightGlue 검증 추가

### 6.3 평가지표

| 목적 | 지표 | 해석 |
| --- | --- | --- |
| 후보 포함 여부 | Recall@1, @3, @5 | 검증된 동일 지점 또는 동일 구역이 상위 K개에 포함되는 비율 |
| 순위 품질 | MRR, nDCG@5 | 여러 단계의 관련도를 고려한 후보 순서 품질 |
| 잘못된 확신 방지 | Top-1 오추천률, 판정 보류율, 점수 구간별 실제 적중률 | 낮은 근거를 강한 일치로 표현하지 않는지 점검 |
| 근거 신뢰성 | 출처 연결 근거율 | 화면의 각 근거가 공식 기록 또는 검증된 메타데이터와 연결되는 비율 |
| 운영 가능성 | CPU/GPU p50·p95 지연, VRAM/RAM, 캐시 생성 시간 | 시현 환경에서의 응답성 |
| 강건성 | 흑백화, 저해상도, 크롭, 조도 변화별 Recall@K | 과거 기록 사진의 열화에 대한 안정성 |

가중치와 보류 임계값은 검증 세트에서만 고르고, 최종 평가는 별도 홀드아웃 세트로 수행한다. 모든 실험 결과는 `benchmarks/location_matching/`의 CSV와 JSON으로 기록하며, UI에는 노출하지 않는다.

### 6.4 재현 기록

각 실험 행에는 질의 사진 ID, 후보 인덱스 버전, 모델 ID와 해시, 디바이스, 실행 시각, 가중치, GPS 정확도, 태그 입력, 후보 순위, 정답 라벨, 지연 시간을 저장한다. 이 기록으로 같은 입력에서 같은 후보 순서가 나오는지 확인한다.

## 7. API와 화면 이식 범위

다음 구현 단계에서 `POST /api/v1/location-matches`가 사진, GPS, 정확도, 장소 단서를 받아 후보 목록과 근거를 반환한다. API 응답에는 원시 모델 점수와 정규화 점수를 분리해 기록하지만, 사용자 화면에는 근거형 요약과 후보 검색 점수만 사용한다.

현재 단계의 범위는 보고서, 특징 캐시, 실험 프로토콜까지다. 실제 후보 검색 API와 UI는 좌표·태그 메타데이터와 실험 결과가 준비된 뒤 연결한다.
