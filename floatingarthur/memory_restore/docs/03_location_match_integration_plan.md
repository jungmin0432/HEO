# AI 시간제작소 위치-기록 연결 기능 이식 계획서

## 0. 문서 목적과 현재 완료 상태

이 문서는 `현재 사진과 과거 을지로 기록 사진을 연결하기` 기능을 기존 을지로 기억복원소에 이식하기 위한 실행 계획서다. 사진 복원 기능을 대체하지 않고, 사용자가 복원 기준 기록을 찾는 앞단 서비스로 추가한다.

완료된 선행 산출물:

- `docs/02_location_match_service_concept.md`: 문제 정의, 모델 비교, 성공 기준, 실험 계획
- `cache/location_matching/`: 공개 기록 사진 56장의 CLIP 및 DINOv2 재사용 특징
- CLIP `(56, 512)`, DINOv2 `(56, 384)`, attention `(56, 16, 16)`, 모델별 `(56, 56)` cosine 행렬

아직 이식하지 않은 범위:

- 현재 사진 질의 API
- GPS·랜드마크 기반 후보 축소
- LightGlue 국소 특징 검증
- 사용자 UI와 실제 실험 결과

## 1. 문제와 이식 목표

### 1.1 해결할 사용자 문제

방문자는 현재 장소의 사진을 남기지만, 그 장면과 연결되는 과거 공개 기록을 찾기 어렵다. 운영자는 기록 사진 56장을 직접 비교해야 하며, 현재 데이터에는 좌표·랜드마크가 구조화된 사진이 0장이다.

### 1.2 이식 목표

1. 사진, GPS 좌표, GPS 정확도, 주변 장소 단서를 받아 관련 기록 사진 상위 후보를 반환한다.
2. 후보별로 시각적 구조, 위치 근접, 장소 단서, 기록 출처, 한계를 분리해 보여준다.
3. 근거가 부족하면 최상위 후보를 정답처럼 제시하지 않고 `판정 보류`로 처리한다.
4. 이후 복원 요청에는 사용자가 선택한 기록 ID만 전달해 기존 복원 API와 연결한다.

## 2. 목표 구조

```text
현재 사진 + 위도/경도 + GPS 정확도 + 주변 장소 입력
                 |
                 v
       위치·랜드마크 후보 축소
                 |
                 v
    CLIP + DINOv2 캐시 기반 후보 재정렬
                 |
                 v
        상위 3~5개 국소 특징 검증
                 |
                 v
 후보 목록 + 근거 + 출처 + 판정 보류 여부
                 |
                 v
      사용자가 기록 선택 -> 기존 복원 API
```

## 3. 데이터 이식

### 3.1 새 인덱스

`data/archive_index.json`을 새로 만든다. 기존 `places.json`은 복원 시연용 장소 카드 역할을 유지한다.

필수 필드:

```json
{
  "asset_id": "archive-image-id",
  "relative_path": "history_photo/.../image.jpg",
  "archive": {"title": "", "source_url": "", "year": 1983},
  "zone": "euljiro-3ga",
  "location": {"latitude": null, "longitude": null, "radius_m": null, "source_url": null},
  "landmark_tags": [],
  "verification_level": "REFERENCE"
}
```

입력값은 공공 기록 원문, 서울시·서울시설공단의 공간 안내, 검증 가능한 지도 자료로만 채운다. 확인하지 못한 좌표는 `null`로 두며 추정값을 사실 좌표로 저장하지 않는다.

### 3.2 특징 캐시

`cache/location_matching/manifest.json`의 SHA-256과 `archive_index.json`의 `relative_path`를 대조한다. 기록 사진이 추가·교체되면 캐시 생성 스크립트를 다시 실행한다.

## 4. API 이식 계획

### 4.1 후보 검색

`POST /api/v1/location-matches`

Multipart 입력:

| 필드 | 필수 | 설명 |
| --- | --- | --- |
| `photo` | 예 | JPG, PNG, WebP 현재 사진 |
| `latitude`, `longitude` | 아니오 | 사용자가 입력한 좌표 |
| `gps_accuracy_m` | 아니오 | 좌표 정확도. 큰 값일수록 위치 가중치 축소 |
| `landmark_text` | 아니오 | 역, 출입구, 주변 건물, 통로 등 |
| `limit` | 아니오 | 반환 후보 수. 기본 5, 최대 10 |

응답 핵심 필드:

```json
{
  "decision": "CANDIDATES | HOLD",
  "model_version": "clip... + dino... + index...",
  "candidates": [{
    "asset_id": "...",
    "retrieval_score": 0,
    "confidence_band": "high | medium | low",
    "evidence": {
      "visual_structure": "...",
      "location_proximity": "...",
      "landmark_consistency": "...",
      "local_feature_consistency": "..."
    },
    "limitations": ["..."],
    "archive": {"source_url": "..."}
  }]
}
```

원시 CLIP·DINO cosine, 가중치, 상세 벡터 값은 실험 로그에는 남기지만 사용자 화면의 API 계약에는 노출하지 않는다.

### 4.2 오류와 보류

- 잘못된 좌표 형식, 손상된 사진, 캐시·인덱스 버전 불일치는 400 또는 409으로 명확히 반환한다.
- GPS 누락은 오류가 아니라 이미지·장소 단서 모드로 전환한다.
- 후보의 공식 위치 메타데이터가 없으면 위치 근거를 `확인 불가`로 표시한다.
- 상위 점수가 임계값 미만이거나 근거가 충돌하면 `HOLD`를 반환한다.

## 5. 서비스 화면 이식

기존 `기록 사진 복원 / 나의 사진 추가` 다음에 `현재 장소에서 기록 찾기` 모드를 추가한다.

1. 사진 업로드
2. 위도·경도와 GPS 정확도 입력, 또는 입력 건너뛰기
3. 주변 장소 입력
4. 후보 3~5개 비교
5. 근거와 한계 확인
6. 기록 선택 후 기존 `복원 시작` 단계로 이동

화면에서 지켜야 할 문구 원칙:

- `동일 장소입니다` 대신 `연결 가능성이 있는 기록 후보`
- `87% 일치` 대신 `후보 검색 점수 87/100`
- 모든 후보에 출처 링크와 검증 수준 표시
- `판정 보류`를 실패 화면이 아니라 다음 탐색 행동으로 제시

## 6. 모델 런타임 이식

### 6.1 단계별 모델

| 단계 | 구성 | 실행 시점 |
| --- | --- | --- |
| 1차 검색 | DINOv2 ViT-S/14, CLIP ViT-B/32 | 요청 사진 1장만 임베딩 |
| 후보 축소 | GPS 거리 감쇠, 랜드마크 태그 | 인덱스 조회 |
| 2차 검증 | DISK + LightGlue | 상위 3~5개만 실행 |

서버 시작 시 캐시 행렬과 인덱스를 메모리에 읽는다. 모델은 지연 로딩하고, GPU가 없으면 CPU 모드로 동작한다. 국소 특징 검증은 CPU 지연이 기준을 넘으면 비활성화하고 `검증 생략` 근거를 반환한다.

### 6.2 가중치 확정 방법

가중치는 코드에 고정하기 전에 `benchmarks/location_matching/`의 검증 세트에서 선택한다. 비교군은 DINOv2만, CLIP만, 두 모델 앙상블, 장소 단서 결합, GPS 결합, LightGlue 추가 순으로 고정한다.

## 7. 평가 기준 반영 실험

### 7.1 사전 성공·실패 기준

| 항목 | 통과 기준 | 미달 시 조치 |
| --- | --- | --- |
| 단일 모델 대비 | Recall@3이 최선 단일 모델보다 10%p 이상 향상 | 가중치 또는 후보 축소 재설계 |
| 공간 단서 효과 | nDCG@5가 0.05 이상 향상, Top-1 오추천률 악화 없음 | GPS·태그 비중 축소 |
| 강한 추천 | 상위 점수 구간의 동일 지점/동일 구역 적중률 80% 이상 | 해당 구간을 HOLD로 전환 |
| 운영 가능성 | 캐시 사용 p95 8초 이하, API 오류율 1% 이하 | 모델 축소 또는 2차 검증 생략 |
| 근거 신뢰성 | 화면 근거 출처 연결률 100% | 출처 없는 문장 제거 |

### 7.2 재현 파일

- `benchmarks/location_matching/experiment_template.csv`: 한 질의당 입력, 모델 버전, 가중치, 순위, 라벨, 지연 기록
- `benchmarks/location_matching/`: 실행별 CSV, 집계 JSON, 모델·인덱스 해시
- `tests/test_location_matching.py`: 입력 검증, 캐시-인덱스 정합성, HOLD 규칙, 응답 계약

## 8. 구현 순서와 완료 조건

1. `archive_index.json` 초안 작성 및 공식 출처 검증
2. 캐시 로더와 후보 점수 서비스 작성
3. API 계약 및 오류 처리 테스트
4. 현재 사진 검색 UI와 후보 근거 화면 추가
5. 20장 이상 독립 질의·라벨로 비교 실험
6. 성공 기준 통과 시 기존 복원 흐름 연결

각 단계는 결과 파일과 테스트를 남긴 뒤 다음 단계로 이동한다. 5단계의 성공 기준에 미달하면 UI 연결 전에 가중치·데이터·보류 규칙을 수정한다.

## 9. 한계와 금지 주장

- 기록 사진과 현재 사진의 시점·시야·조도 변화가 크므로 이미지 유사도는 위치 증명이 아니다.
- 현재 GPS는 지하에서 불안정할 수 있으므로 단독 판정 근거가 아니다.
- 좌표·랜드마크 메타데이터가 없는 기록은 정확한 위치 후보라고 표현하지 않는다.
- 실험 전에는 모델의 위치 정확도나 상권 효과를 주장하지 않는다.

## 10. 담당자 인수 기준

다음 담당자는 이 문서, 2차 서비스 구상 보고서, 특징 캐시 manifest, 실험 템플릿만으로 API와 UI 작업을 시작할 수 있어야 한다. 모델 가중치 재다운로드가 필요하면 `scripts/build_location_feature_cache.ps1 -AllowDownload`을 최초 한 번만 실행하고, 이후에는 오프라인 캐시 생성으로 재현한다.
