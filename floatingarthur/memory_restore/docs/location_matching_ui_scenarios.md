# 모바일 UI 시나리오: 현재 장면에서 기록 찾기

## 1. GPS와 장소 단서가 있는 방문자

1. 홈의 `현재 장면으로 기록 찾기`를 누른다.
2. 현재 사진을 고르고 `을지로3가역`, `지하 연결 통로` 같은 주변 단서를 입력한다.
3. GPS 위도·경도와 기기 정확도 60m를 선택 입력한다.
4. 후보 목록에서 연도·제목·후보 적합도와 짧은 이유를 비교한다.
5. 상세 화면에서 사진 구조, 입력 위치, 주변 단서, 국소 검증의 미실행 여부와 공개 기록 출처를 읽는다.
6. 스스로 적절하다고 판단한 기록을 골라 복원 화면으로 이동한다.

성공은 자동 확정이 아니라, 사용자가 출처가 있는 후보와 판단 한계를 보고 복원 맥락을 선택할 수 있는 상태다.

## 2. 지하에서 GPS가 없는 방문자

1. 사진과 장소 단서만 입력한다.
2. 목록은 이미지와 장소 태그 중심으로 후보를 정렬한다.
3. 상세 화면은 위치 근거가 비어 있음을 그대로 표시한다.
4. 사용자는 후보를 비교하거나 다시 입력한다.

GPS 부재 때문에 사진 업로드를 막지 않는다. 단, 위치 근거가 약해졌다는 사실은 숨기지 않는다.

## 3. 근거가 부족한 방문자

1. 사진만 입력한다.
2. API가 `HOLD`를 반환하면 결과 화면의 문구는 ‘특정 기록을 추천하지 않음’이 된다.
3. UI는 주변 역명·출입구·통로 또는 GPS 입력을 제안하고, 복원 연결 행동은 제공하지 않는다.

이 흐름은 모델이 애매한 장면을 강하게 연결하는 오류를 줄이기 위한 설계다.

## 4. 후보를 복원으로 이어 가는 방문자

1. 후보 상세의 `이 기록과 내 사진 이어보기`를 누른다.
2. 현재 사진은 원본으로 유지하고, 선택한 공개 기록의 `asset_id`만 복원 API에 보낸다.
3. 결과 화면에는 원본·보수 보정·AI 복원을 나란히 두고, 별도 ‘기록 연결’ 블록에 공개 기록 출처와 매칭 한계를 표시한다.
4. 결과 이미지는 저장할 수 있지만, AI 복원 결과가 과거의 사실을 재현한다는 표현은 사용하지 않는다.

## 5. 모델 준비 또는 통신 오류

1. 후보 API가 503을 반환하면 입력 화면에 오류를 표시한다.
2. 사용자는 사진 복원 기능으로 돌아갈 수 있다.
3. 화면은 모델 세부명, 임베딩 점수, 실험 수치를 노출하지 않는다.

운영자는 `/api/v1/location-matching/status`와 [location_matching_guide.md](location_matching_guide.md)를 통해 캐시와 런타임을 점검한다.
